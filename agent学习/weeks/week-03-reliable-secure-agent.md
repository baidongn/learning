# 第 3 周：可靠、安全、可评测的 Agent

**本周 6–8 小时｜先证明可靠，再做平台**

前置：Agent 已有检索、工具、MCP、审批和恢复。本周把“感觉不错”变成可重复证据，并处理模型、数据和工具失败。

## 学习结果

- 用 Trace 定位 Prompt、路由、检索、工具还是业务数据错误。
- 建立离线/在线评测和回归门禁，评答案也评执行轨迹。
- 处理 Injection、PII、内容审核、危险工具和知识库污染。
- 设计会话、记忆、应用、Prompt/配置版本的数据模型。
- 对 DeepSeek 调用实施超时、限流、重试、熔断、缓存和降级。

## 1. Trace 与调试 `[最小实验]`

LangSmith Trace 应串起一次运行的模型、检索、工具、节点、异常和元数据。每条 Trace 至少关联：`tenant/user/thread/request`、Agent/Prompt/模型版本、节点、延迟、Token、费用和最终状态。

错误归因顺序：

```text
输入是否清楚
→ 路由是否正确
→ 正确资料是否召回/排序
→ 工具名称与参数是否正确
→ 工具业务数据是否正确
→ 模型是否依据 Observation 回答
→ 是否正确终止
```

不要把 Key、完整 PII、未授权文档正文写入 Trace。生产可用采样降低成本，但错误和高风险操作应完整保留审计元数据。

Prompt 版本和代码版本必须同时记录；否则无法复现实验。用户反馈关联具体 Run，而不是只保存“赞/踩”。

## 2. Dataset 与评测 `[基础必会]`

评测集来自真实业务：高频、边界、历史故障、高风险、对抗输入。每条 Case 保存输入、期望行为、允许答案范围、期望来源、期望/禁止工具、风险等级和标签。

四类评测互补：

- **规则评测**：Schema、引用域名、工具序列、PII、延迟；快且稳定。
- **人工评测**：复杂语义与业务可用性；贵但权威。
- **LLM-as-Judge**：规模化语义评分；需要明确 Rubric、参考答案、偏差校准。
- **业务结果**：解决率、转人工率、错误操作率；最接近价值但归因较难。

离线评测用于发布前比较；在线评测用于抽样监控；回归测试对每次 Prompt/模型/工具变化运行。不要用同一个 Judge 替代所有指标。

## 3. 评什么 `[项目实现]`

| 层 | 指标 |
|---|---|
| 答案 | 正确率、完整性、拒答质量、格式通过率 |
| RAG | Recall@K、NDCG、Groundedness、引用准确率/覆盖率 |
| Tool | 选择准确率、参数正确率、成功率、重复调用率 |
| 轨迹 | 执行顺序、审批、停止行为、预算超限率 |
| 系统 | P50/P95 延迟、Token、费用、失败率、可恢复率 |
| 业务 | 首次解决率、转人工率、用户反馈、错误动作数 |

正确引用不只是“有 URL”，而是来源真实存在、确实支持对应陈述、用户有权限访问。最终项目评测器用确定性规则示范答案/引用/轨迹三层；LangSmith 实验是外层能力。

发布门禁示例：核心集正确率不得下降；高风险错误为 0；工具参数成功率 ≥ 99%；P95/成本不得超过预算。具体阈值按业务设定。

## 4. Prompt Injection 与不可信数据 `[基础必会]`

攻击可来自用户、网页、知识库、Tool 返回或 MCP Resource。分层防护：

1. 系统规则与数据明确分隔。
2. 外部内容只作为资料，不允许改变工具权限。
3. Tool Registry 执行独立权限、参数和风险校验。
4. 限制可访问域名、文件、数据库和网络。
5. 输出前检查数据泄露和危险内容。
6. 对知识入库做来源、权限、恶意指令和版本审核。

不要依赖一条“忽略恶意指令”的 Prompt。安全边界必须在模型外。

## 5. PII、审核与租户隔离 `[项目实现]`

- 输入审核：长度、类型、Injection、内容风险和文件安全。
- 输出审核：PII、违规内容、越权事实和引用。
- 流式审核：缓冲小片段再输出；发现风险立即停止并发终止事件。
- 危险工具：最小权限、人工确认、二次校验、幂等和审计。
- 租户隔离：数据库行、向量检索、缓存 Key、对象存储路径、Trace 都带 tenant。

PII 数据最小化：只收业务必需字段，展示脱敏，日志不记录，存储加密，设保留期限和删除接口。

## 6. 记忆治理 `[基础必会]`

写记忆前回答：是否有长期价值？是否敏感？用户是否同意？何时过期？来源是什么？如何删除？

建议字段：`tenant/user/key/value/source/consent/version/created_at/expires_at/deleted_at`。记忆抽取由结构化模型生成候选，再经规则筛选。修改或删除后同步缓存和向量索引。

摘要不是事实数据库；订单状态等动态信息下次应重新调用工具。

## 7. PostgreSQL、SQLAlchemy、Alembic `[最小实验]`

最小数据域：

- `users/tenants`：身份与租户。
- `applications/application_versions`：Agent 与发布配置。
- `prompt_versions`：模板、变量、状态。
- `threads/messages/checkpoints`：会话与恢复。
- `memories`：长期记忆和同意/过期。
- `documents/chunks`：知识与版本/权限。
- `tool_runs/approvals`：轨迹与危险操作审计。
- `api_keys/usage_events`：开放接口与统计。

SQLAlchemy 管理模型和事务；Alembic 管理 Schema 变化。索引围绕真实查询：tenant+user、thread+time、document+version、api_key_hash、usage time。API Key 只存哈希或安全派生值。

Pydantic v2 处理 API/工具契约，不替代数据库约束。唯一约束是幂等最后防线。

## 8. DeepSeek 可靠性 `[项目实现]`

```text
客户端总超时 + 单次连接/读取超时
→ 对 429/网络/部分 5xx 指数退避并加抖动
→ 连续失败触发熔断
→ 可缓存的只读结果按版本缓存
→ 降级到简短回答/只读 RAG/人工转接
```

限流可按 tenant/user/API Key 分层。并发限制保护模型和下游工具。模型请求要记录 Token 和结束原因。结构化输出校验失败只有限重试，重试 Prompt 中提供错误摘要，不发送敏感栈。

Fake Client 是必需测试基础设施，不是临时 Mock：覆盖正常回答、Tool Call、无效 JSON、限流、超时和流式中断。

## 9. 多模型与本地模型 `[理解选型]`

业务只依赖 `ModelGateway`。多模型路由评估能力、工具兼容、上下文、延迟、费用、合规和可用区。课程只运行 DeepSeek；Hugging Face 用于查模型/数据/评测资源，本地模型用于数据主权或成本场景。不要在没有评测集时切模型。

## 综合练习

1. 从历史问题整理 20 条 Dataset，至少 5 条高风险/对抗输入。
2. 为答案、引用、工具参数和终止行为各写一个规则评测。
3. 修改一个 Prompt，比较基线与候选，不只看平均分。
4. 构造知识库 Injection，验证不能触发工具。
5. 测试两个用户的记忆、检索和 Thread 不互串。
6. 让 Fake Client 返回无效结构、超时、重复 Tool Call，验证回退。

## 常见错误

- 只评答案相似度，不评工具轨迹。
- Dataset 全是简单正例。
- Judge 没有 Rubric，也不和人工校准。
- Trace 记录全部敏感正文。
- 租户隔离只写在 Prompt。
- 模型失败无限重试，形成重试风暴。

## 本周验收

- [ ] 一次错误能定位到具体层。
- [ ] 评测可重复运行并比较版本。
- [ ] 错误引用会被判失败。
- [ ] Injection 不能改变工具权限。
- [ ] 危险操作有确认和审计。
- [ ] 记忆可查看、过期、删除且用户隔离。
- [ ] DeepSeek 超时/限流有有限重试和降级。
